# 万枢物流 — 技术选型方案

> 本文档基于万枢物流四端（管理后台、用户端、快递员端、司机端）的业务需求，给出务实精简的技术选型方案。每项技术均对应明确的业务场景，不为用而用。

## 二、认证与权限方案

| 端 | 登录方式 | 权限控制 | 技术实现 |
|---|---|---|---|
| **用户端（小程序）** | 微信小程序登录（`wx.login`） | 无 RBAC 权限，仅登录态校验 | Sa-Token `StpUtil.login()` 签发 Token |
| **快递员端** | 账号 + 密码 | Sa-Token RBAC 角色权限 | 角色：`courier`，绑定取件/派件/消息等权限 |
| **司机端** | 账号 + 密码 + 短信验证码 | Sa-Token RBAC 角色权限 | 角色：`driver`，绑定提货/在途/交付等权限 |
| **管理后台** | 账号 + 密码 + 图形验证码 | Sa-Token RBAC 菜单 + 按钮级权限 | 角色：`admin` 等多级角色，树形权限配置 |

**关键设计**：
- 四端共用一套后端 API，通过 Sa-Token 的**多账号体系** (`StpUserUtil` / `StpUtil`) 区分用户端与管理端登录态
- 用户端仅做身份校验，不做菜单/按钮权限拦截
- 管理后台采用 **角色 → 权限 → 菜单/按钮** 三级权限模型

## 三、数据库选型

### 3.1 MySQL 8 — 核心业务主库

| 存储内容 | 说明 |
|---|---|
| 用户/账号 | 用户端用户、快递员、司机、管理员 |
| 机构体系 | 总公司 → 分公司 → 转运中心 → 分拣中心 → 营业部 |
| 运费模板 | 同城寄、省内寄、跨省寄、经济区互寄 |
| 车型/车辆 | 车辆基本信息、行驶证、绑定司机 |
| 订单/运单 | 核心业务单据，强事务一致性 |
| 线路/车次 | 干线、支线、专线、临时线路、接驳路线 |
| 取派件作业 | 快递员取件/派件任务分配 |
| 排班 | 快递员/司机排班管理 |
| 角色/权限 | RBAC 权限数据 |

**选择理由**：订单/运单等核心业务对事务一致性要求高，关系查询复杂，MySQL 是最成熟稳定的选择。

### 3.2 Redis 7 — 缓存与会话

| 使用场景 | 数据结构 | 说明 |
|---|---|---|
| 登录会话（Token） | String | Sa-Token 会话存储 |
| 图形验证码 | String + TTL | 管理后台登录验证码，60s 过期 |
| 短信验证码 | String + TTL | 司机端登录验证码，5min 过期 |
| 热点数据缓存 | Hash | 机构树、运费模板等低频变更高频读取数据 |
| 分布式锁 | String + NX | 调度任务防并发、运单状态变更防重复 |
| 工作台统计缓存 | String | 今日订单量/金额等实时统计数据缓存 |

### 3.3 MongoDB — 轨迹与日志

| 使用场景 | 说明 |
|---|---|
| **物流轨迹** | 运单流转记录（到达XX转运中心/XX营业部/已签收等），每单轨迹节点数不固定，文档模型天然适配 |
| **操作日志** | 管理端操作日志（谁在什么时间、什么IP、操作了什么），写入量大，无需复杂事务 |
| **异常日志** | 系统运行报错信息，结构不固定 |
| **运单状态变更历史** | 运单从"待调度→已调度→运输中→已签收"全链路状态快照 |

**选择理由**：轨迹和日志数据具有写入量大、schema 不固定、不需要跨表事务的特点，MongoDB 文档模型比 MySQL 更高效灵活。

### 3.4 Neo4j — 物流网络图

| 使用场景 | 说明 |
|---|---|
| **机构层级关系** | 总公司→分公司→转运中心→分拣中心→营业部，树/图结构关系查询 |
| **线路网络拓扑** | 干线/支线/专线组成的全国物流网络图，节点=机构，边=线路 |
| **最优路径计算** | 给定寄件地址和收件地址，在线路图中计算转运次数最少或成本最低的路径 |
| **调度辅助查询** | 查询某转运中心的上下游关联机构、某线路的替代线路等图遍历操作 |

**选择理由**：物流网络天然是一张有向加权图。调度配置中明确要求"转运次数最少"和"成本最低"两种路径优化策略，这正是图数据库的核心优势。用 MySQL 递归查询实现同类需求性能差且代码复杂。

**数据同步策略**：MySQL 为主数据源，机构和线路数据变更时通过 RabbitMQ 异步同步到 Neo4j。

## 四、中间件选型

### 4.1 RabbitMQ — 异步消息队列

| 使用场景 | Exchange 类型 | 说明 |
|---|---|---|
| **订单状态变更通知** | Topic | 用户下单后通知快递员端、管理后台更新待办 |
| **运单调度异步处理** | Direct | 运单创建后异步匹配最优路径和车次 |
| **消息推送** | Fanout | 公告、系统通知推送到快递员端消息中心 |
| **数据同步** | Direct | MySQL 机构/线路变更 → Neo4j 图同步 |
| **超时告警** | Dead Letter | 运输任务超时未完成自动触发告警 |

**选择理由**：物流系统涉及大量异步流程（下单→调度→分配→取件→运输→派件→签收），RabbitMQ 成熟稳定，Spring Boot 集成方便，消息确认机制保证可靠投递。

### 4.2 阿里云 OSS — 对象存储

| 使用场景 | 说明 |
|---|---|
| **提货凭证** | 司机提货时上传的提货单照片 |
| **交付凭证** | 司机交付时上传的货物照片和交货单照片 |
| **行驶证/驾驶证** | 车辆管理和司机管理中的证件图片 |
| **头像** | 用户和快递员头像 |
| **电子存根** | 运单电子存根 |

**接入方式**：
- 后端集成阿里云 OSS Java SDK
- 前端通过后端获取 STS 临时凭证，直传 OSS（减轻服务器带宽压力）
- 图片上传后返回 CDN 加速的访问 URL

### 4.3 XXL-JOB — 分布式任务调度

| 定时任务 | 执行周期 | 说明 |
|---|---|---|
| **超时订单处理** | 每 5 分钟 | 检测超过最晚派单时间的运单，标记为超时 |
| **运输任务超时检测** | 每 10 分钟 | 检测超过最晚送达时间的运输任务，推送告警 |
| **工作台数据统计** | 每小时 | 汇总各机构订单量、金额、任务完成率 |
| **全国订单分布统计** | 每日凌晨 | 按省份汇总订单分布数据（供工作台地图展示） |
| **数据归档** | 每周 | 将已完成超过 30 天的运单轨迹数据归档 |

## 五、实时通信

### 5.1 SSE (Server-Sent Events) — Coze AI 流式输出

| 使用场景 | 说明 |
|---|---|
| **智能客服回复** | 用户端/快递员端对话时，Coze 的回答通过 SSE 流式返回，实现逐字输出效果 |
| **调度分析报告** | 管理后台请求 Coze 生成运营分析时，流式展示生成过程 |

**技术实现**：
- 后端通过 `SseEmitter`（Spring Boot 内置）将 Coze API 的流式响应转发到前端
- 前端使用 `EventSource` API 接收流式数据
- 小程序端使用 `uni.request` 的 `enableChunked` 模式接收

### 5.2 WebSocket — 实时消息推送

| 使用场景 | 说明 |
|---|---|
| **物流轨迹实时更新** | 司机端/快递员端操作后，管理后台运单详情页实时刷新轨迹 |
| **订单状态变更推送** | 用户端查快递页面，运单状态变更时实时推送通知 |
| **消息中心实时提醒** | 快递员端新取件/派件任务到达时，实时弹出提醒 |
| **工作台待办更新** | 管理后台工作台页面，待办任务数量实时刷新 |

**技术实现**：
- 后端使用 Spring Boot 内置的 `spring-boot-starter-websocket`
- 管理后台（Vue 3）直接使用原生 WebSocket API
- 小程序端使用 `uni.connectSocket` API
- 配合 Redis Pub/Sub 实现多实例间消息广播

## 六、Coze AI 集成方案

### 6.1 智能客服（用户端 / 快递员端）

在小程序端嵌入 AI 客服入口，用户可通过自然语言对话完成以下操作：

| 能力 | 示例对话 | 实现方式 |
|---|---|---|
| **物流查询** | "我的快递到哪了？单号 WS20240101001" | 调用运单查询 API，格式化返回轨迹 |
| **运费估算** | "从北京寄到上海 5 公斤大概多少钱？" | 调用运费计算 API，返回估算价格 |
| **寄件指引** | "我想寄一箱书到广州" | 引导填写寄件表单，推荐物品分类 |
| **常见问题** | "禁止寄递哪些物品？" | 知识库检索，返回禁寄物品目录 |
| **取件进度** | "快递员什么时候来取件？" | 查询取件任务状态，返回预计时间 |

### 6.2 智能调度辅助（管理后台）

在管理后台嵌入 AI 助手面板，辅助运营人员决策：

| 能力 | 示例对话 | 实现方式 |
|---|---|---|
| **异常订单分析** | "今天有哪些超时订单？原因是什么？" | 查询超时运单，结合轨迹分析瓶颈环节 |
| **运营数据总结** | "本周运营情况怎么样？" | 汇总订单量/完成率/超时率，生成自然语言报告 |
| **路线优化建议** | "北京到成都走哪条线路更划算？" | 查询 Neo4j 线路图，对比成本和时效 |
| **调度建议** | "明天有 200 单需要配送，怎么安排快递员？" | 结合排班和作业范围，给出分配建议 |

### 6.3 Coze 接入架构

```
前端 ──SSE──> 后端 (SseEmitter) ──HTTP Stream──> Coze API
                    │
                    ├── Coze Plugin: 运单查询
                    ├── Coze Plugin: 运费计算
                    ├── Coze Plugin: 轨迹查询
                    ├── Coze Plugin: 订单统计
                    └── Coze 知识库: 禁寄物品/常见问题/业务规则
```

**关键配置**：
- 在 Coze 平台创建 Bot，配置 Plugin 对接后端 API
- 后端封装 Coze OpenAPI 调用，通过 SSE 流式转发
- Coze Bot 的 System Prompt 中注入物流业务上下文
